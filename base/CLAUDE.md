## Architecture (do not restructure)

```
src/app/
├── core/       # App-wide singletons: guards, interceptors, root services. Never imported by features circularly.
├── shared/     # Reusable, presentation-only UI. No business logic, no services with side effects.
├── features/   # Domain-specific, lazy-loaded feature areas. One folder per feature.
├── layout/     # Shell: header, footer, nav.
├── app.config.ts
└── app.routes.ts
```

New features go in `features/<feature-name>/`. Do not add top-level folders outside this
structure without asking first.

## Hard constraints (enforced by hooks — listed here for visibility, not as the only defense)

1. **Do not add, remove, or upgrade dependencies** outside what's already in `package.json`
   for this project's selected bundles. If a task seems to need a new package, stop and
   flag it — do not `npm install` it yourself.
2. **Do not edit** `angular.json`, `.github/**`, `package-lock.json`, or any `.env*` file.
3. **Do not touch files under `core/`** (guards, interceptors, auth) unless the task is
   explicitly about authentication/authorization and the relevant bundle rule file says
   it's expected.
4. **Follow the Angular conventions in `.claude/rules/angular.md`** — these are Angular's
   own official guidance, not a house preference.
5. **Follow the bundle-specific rules** in `.claude/rules/` for whichever options were
   selected (auth, data layer, state, roles, deploy target) — each explains the pattern
   already chosen for this project. Do not introduce a different pattern for the same
   concern (e.g. do not add a second HTTP client approach if one is already defined).
