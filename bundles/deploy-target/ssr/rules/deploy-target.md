# Deploy Target: SSR (Server-Side Rendered)

## What pattern is used

This project uses Angular's own server-side/hybrid rendering (`@angular/ssr`), added by
running the real Angular CLI schematic (`ng add @angular/ssr`) at generation time —
**not** hand-written server bootstrap files. This keeps the SSR plumbing (`server.ts`,
`app.config.server.ts`) exactly matched to whatever Angular version this project
actually uses, since that plumbing changes across Angular versions.

Per-route rendering strategy is controlled in `src/app/app.routes.server.ts`, using
`RenderMode.Client` (browser-rendered), `RenderMode.Prerender` (built once, served
static — for pages with no per-request dynamic data), or `RenderMode.Server`
(rendered per-request on the server — for pages needing per-request/user-specific
data).

## What the AI agent may do

- **Add an entry to `app.routes.server.ts` for every new route added to
  `app.routes.ts`.** This file is expected to grow as routes are added — it is not
  protected. Choose the render mode deliberately:
  - `RenderMode.Prerender` for static/marketing pages with no per-request data.
  - `RenderMode.Server` for pages needing request-specific data (e.g. a logged-in
    user's dashboard).
  - `RenderMode.Client` for pages that genuinely need to skip SSR (e.g. heavy
    client-only interactivity with no SEO value).
- Use the SSR-specific tokens from `@angular/core` (`REQUEST`, `RESPONSE_INIT`,
  `REQUEST_CONTEXT`) inside a component/service if a feature genuinely needs access to
  the request/response during server rendering (e.g. reading a cookie, setting a
  status code) — these are the documented, supported way to do this, not a workaround.

## SSRF protection (production deployment concern)

Angular's Node SSR engine validates `Host`/`X-Forwarded-*` headers by default to prevent
server-side request forgery (SSRF) via host-header injection — by default, all
`X-Forwarded-*` headers are treated as untrusted and stripped. If this project is
deployed behind a reverse proxy or load balancer that needs these headers trusted, that
requires explicit configuration in `src/server.ts` (`allowedHosts`, `trustProxyHeaders`
on `AngularNodeAppEngine`) — a deployment/infra decision, not something to configure
speculatively.

- **Never set `allowedHosts: ['*']`** to make a "host not recognized" error go away —
  this defeats the protection entirely and is explicitly called out as a security risk
  in Angular's own docs. If a real deployment needs multiple hostnames allowed, list
  them explicitly (wildcards like `*.example.com` are fine — a bare `*` is not).
- **Do not set `trustProxyHeaders: true` (trusting all `X-Forwarded-*` headers)**
  without confirming this project is actually behind a proxy that sets and validates
  them — an untrusted deployment trusting these headers is itself an SSRF risk.
- Since `src/server.ts` is protected (see above), any change to `allowedHosts` /
  `trustProxyHeaders` goes through a developer, not through this agent directly — flag
  the need rather than editing the protected file.

## What the AI agent must NOT do

- **Do not edit** `src/server.ts` or `src/app/app.config.server.ts` — protected by
  `.claude/settings.json`. These were generated by the Angular CLI schematic and match
  this project's exact Angular version; hand-editing them risks breaking that alignment.
- Do not add a new route to `app.routes.ts` **without** adding a matching entry to
  `app.routes.server.ts` — an unmapped route falls through to whatever the catch-all
  (`**`) render mode is, which may not be the right choice for that specific page.
- Do not assume `window`, `document`, or other browser-only globals are available
  without guarding for the server context (`isPlatformServer` from `@angular/common`) —
  code here runs on the server during SSR, not just in the browser.

## Where the code lives

- `src/server.ts` — Node/Express server entry point (protected, CLI-generated).
- `src/app/app.config.server.ts` — server-specific provider config (protected,
  CLI-generated).
- `src/app/app.routes.server.ts` — per-route render mode (editable, expected to grow
  with every new route).
